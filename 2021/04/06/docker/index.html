docker学习

自己的理解，docker就是可以通过镜像，快速的构建需要的软件，例如，，mysql 普通用户需要下载mysql包，然后按照mysql教程去安装部署，但是通过docker 一条命令就可以完全安装好，通过docker 的命令启动

一、docker 扫盲

docker 是利用cgroup 资源分脱机制和linux 核心命名空间namespace 构建独立的容器containers , 是操作系统层面虚拟化技术，由于隔离的进程独立于宿主机和其他隔离的进程，所以又叫做容器，docker在容器的基础上进一步的进行封装，从文件系统，网路互联到进程隔离，简化了容器的创建于维护，使得他能够更轻便，快捷，Docker 可以在单一linux 下运行，避免创建一个虚拟机带来的负担。

docker 与虚拟机的区别与特点

对于咱们这些新手来说，尤其不知道学习docker和建立虚拟机有啥区别，从网络中弄来一张图，来进行一下的对比，

1、传统的虚拟机需要硬件系统，自己的操作系统，虚拟机一旦启动，就需要对资源预占， 每个虚拟机需要必要的app 二进制库以及整个操作系统。

docker 则与宿主机共享硬件资源和内核，与其他容器共享内核，在用户空间以分离的进程运行,容器内没有自己的内核，也没有进行硬件虚拟。

Docker 有什么特点

  • 更快的启动速度 顾名思义 相较于虚拟机，docker 的启动速度在秒级，而虚拟机可能需要几分钟才能启动起来，主要因为直接利用宿主机的内核，无需启动完成的操作系统

  • 更高效的资源利用率 因为docker 不需要进行硬件模拟，docker能够更好的进行资源的利用

  • 更高的系统支持量 docker 的架构能够共享应用程序库和内核，所占的内存极小，同样的硬件环境，docker 运行的容器数量远远大于虚拟机的数量

  • 持续交付与部署 对于开发人员来说，一次创建，多处运行，可以通过定制镜像的方式来实现持续集成交付部署，开发人员可以通过dockerFile 进行镜像的构建，运维人员可以通过镜像进行部署。

  • 更快的迁移 Docker 确保了执行环境的一致性，所以只需要将镜像迁移部署到相关的平台，能够更快的进行迁移

  • 更弱的隔离性 docker 是进程之间的隔离，而虚拟机是系统级别的隔离。

  • 更弱的安全性 docker 一旦从不同用户升级到root用户，他直接具备了宿主机的root权限，可以进行宿主机的操作。

    二、docker 的基本概念

    docker 有三剑客 镜像、仓库、容器

    每个来介绍一下

    • 镜像 docker 是一个文件系统，他包含了容器运行需要的资源，文件，程序和库，还包括程序运行的需要的环境变量和一些匿名卷，但是不包含动态数据，一旦构建就不能修改

    • 容器 容器其实就是个进程，但是跟宿主机的其他进程不一样，是在自己的namespace中独立运行的进行，能够进行 启动，停止，暂停，删除操作

    • 仓库 构建镜像就可在当前宿主机进行发布，构建，部署，但是在其他虚拟机上怎么部署呢，这就像跟gitlab似的，需要将镜像存储到仓库中，然后到别的机器上时需要将这个仓库中的镜像拉取下来进行构建，部署

      Docker Registry 就是这样的服务。一个 Docker Registry 中可以包含多个仓库；每个仓库可以包含多个标签；每个标签对应一个镜像，其中标签可以理解为镜像的版本号。

docke 三剑客来了

docker-compose : 可以理解为有很多的镜像，总不能每次都docker pull 然后 docker run吧，这不符合程序员的思维，所以docker 就整出 来了docker-compose， 其实就是一个.yaml ,在其中书写你程序启动需要的镜像，然后交给docker 进行自动化的构建

​ 这些镜像，并且启动起来。

docker-machine: 这个怎么理解呢？ 前面讲到docker 是利用linux 的cgroup 资源分脱技术，但是如果没有linux 咋办，这就需要能够模拟出来linux的虚拟机来进行docker容器的创建。用于在各种平台上快速创建具有 docker 服务的虚拟机的技术，甚至可以通过指定 driver 来定制虚拟机的实现原理（一般是 virtualbox）。

docker-swarm : 这个就更好理解了，在现代发展中，集群的概念已经很成熟了，docker 怎么能够不支持集群呢，所以docker-swarm 可以通过很简单的命令进行集群的创建。实现分布式服务

docker commit

我们所有的改动都会记录到docker的存储层，docker 可以通过docker commit 将存储层构建成镜像，换句话说就是将存储层

为啥不用docker commit 定制镜像?

但是实际环境中并不推荐如上所示的 commit 用法，具体可以查看《Docker 从入门到实践》中的描述，总结下来为三点：

  1. 由于 commit 命令的执行，有很多文件被改动或添加了。这还仅仅是最简单的操作，如果是安装软件包、编译构建，那会有大量的无关内容被添加进来，如果不小心清理，将会导致镜像极为臃肿。
  2. 使用 commit 生成的镜像也被称为黑箱镜像，换句话说，就是除了制作镜像的人知道执行过什么命令、怎么生成的镜像，别人根本无从得知。
  3. 不符合分层存储的概念，即除当前层外，之前的每一层都是不会发生改变的。
